# 数据采集流程规范（Pipeline Spec）

本项目必须遵循四阶段采集模型，且所有站点统一按此流程设计。

---

## Stage 1：列表页采集（List Extraction）

### 输入
- 列表页 HTML
- 或浏览器已完成渲染的页面

### 目标
提取基础字段与详情 URL，不做复杂详情解析。

### 输出
- 基础字段
- 详情页 URL
- 关联页 URL
- 去重主键或业务主键

### 规则
- 只做结构定位、字段提取、链接标准化
- 不允许在本阶段解析详情富文本块
- 必须保留来源页 URL
- 必须支持分页或筛选上下文记录

---

## Stage 2：详情页采集（Detail Extraction）

### 输入
- Stage 1 产出的详情页 URL / 关联页 URL

### 目标
提取详情字段，并转换为结构化对象。

### 输出
- 文本字段
- 列表字段
- 统计字段
- 关联实体字段

### 规则
- 必须显式处理字段缺失
- 能结构化的内容必须结构化，不能只保留 raw text
- 如需保留 raw text，应作为兜底字段而非唯一字段
- 详情页、关联页解析逻辑必须独立封装

---

## Stage 3：结构化输出（Normalization）

### 目标
将 Stage 1 与 Stage 2 结果汇总为最终 JSON 文件。

### 输出文件
- all.json：层级树结构
- xxx-flat.json：扁平结构
- meta.json：任务元信息
- xxx.partial.json：失败中断时的临时文件（可选但推荐）

### 规则
- 必须统一字段命名
- 必须保留抓取时间
- 必须保留来源 URL
- 必须保证 JSON 可直接入库或二次处理

---

## Stage 4：自动化执行（Automation）

### 运行方式
- 使用 GitHub Actions 手动触发或后续扩展为定时触发

### 每次运行必须完成
1. 安装依赖
2. 安装 Playwright 浏览器
3. 执行脚本
4. 生成 output/
5. 同步 output/ 到仓库 data/ 对应目录
6. 自动 commit & push
7. 上传 artifact 便于排查

### 规则
- 输出目录不存在时必须显式报错或提示
- commit 前必须检查是否有文件变化
- artifact 上传建议 always() 执行

---

## 强制约束
- 不允许跳过 Stage 1 直接硬写详情
- 不允许在 Stage 1 做 Stage 2 的解析工作
- 不允许省略来源 URL
- 不允许只在 Actions 中生成文件而不回写仓库
